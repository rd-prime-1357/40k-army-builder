# 40K Army Builder — Data Dictionary
**Version 2.0 — July 2026**

The unit data is generated by the Python data pipeline (Wahapedia CSV exports → transform → MFM points → integrity check → convert to JSON → loadout parser), not hand-maintained in Excel (Excel as a data-maintenance tool was retired per D5; this supersedes D7). The CSV tables below are the intermediate pipeline artifacts. The merge key across Unit_Stats, Unit_Weapons, Unit_Wargear_Options, and Unit_Other_Options is **Army Name + Unit Name + Model Group**.

---

## Tab 1: Unit_Stats

One row per Model Group per unit. Stores base stat values exactly as printed on datasheets. Does not store modifier values from leaders, enhancements, or detachment rules — those are assembled at runtime by the app.

Save values (SV, INV, FNP) are stored as integers (e.g. 3 for a 3+ save); the "+" is added by the display layer. INV and FNP are treated as explicit stat fields rather than ability text, to support prominent display alongside SV and runtime modifier tracking.

| Field Name | Data Type | Notes |
|---|---|---|
| Army Name | char | e.g. Chaos Daemons. **Merge key.** |
| Unit Name | char | As printed on datasheet. **Merge key.** |
| Model Group | char | "All" for units with one stat line. Named group for units with multiple stat lines (e.g. "Pink Horror", "Blue Horror"). **Merge key.** |
| Unit Type | char | Epic Hero, Character, Battleline, Infantry, Mounted, Beast, Monster, Vehicle, Dedicated Transport, Fortification, Allied. Note: Epic Hero and Character are distinct unit types that share some rules (leader eligibility, warlord eligibility, enhancement eligibility) but are not synonymous. |
| M | char | Move. e.g. 6" |
| T | int | Toughness. |
| SV | int | Armour Save. e.g. 3 for a 3+ save. |
| INV | int | Invulnerable Save. e.g. 4 for a 4+ save. Blank if none. |
| INV_Condition | char | Optional. Free text condition if INV is conditional. e.g. "mortal wounds". Blank if unconditional. |
| FNP | int | Feel No Pain. e.g. 5 for a 5+ save. Blank if none. |
| FNP_Condition | char | Optional. Free text condition if FNP is conditional. e.g. "mortal wounds". Blank if unconditional. |
| W | int | Wounds. |
| LD | int | Leadership. e.g. 6 for a 6+ test. |
| OC | int | Objective Control. |
| Leader Ability Name | char | The name of this unit's leader joining ability as printed on the datasheet. "Leader" for standard leaders. Named ability (e.g. "Heroes of Ultramar") for non-standard joining mechanics. Blank if this unit is not a leader. |
| Leader Eligible Units | char | Comma-separated list of unit names this unit can lead. Blank if Leader Ability Name is blank. |
| Co-Leader Eligible With | char | Comma-separated list of leader unit names that permit this unit to also join a unit they are already attached to. Blank if this unit must be the sole leader in a unit. |
| Leader Restrictions | char | Free text capturing any restrictions on the joining mechanic that deviate from standard leader rules. e.g. "Cannot join an Attached unit." Blank if no non-standard restrictions apply. |
| Unit Ability Names | char | Comma-separated list of parameterized ability names as printed on the datasheet. e.g. "Deadly Demise 1, Scouts 6"". Full generic descriptions in Tab 8: Unit_Abilities. |
| Rule Names | char | Comma-separated list of rule names. Full descriptions in Tab 5: Rules. |
| Keyword Names | char | Comma-separated list of keyword names. Full descriptions in Tab 6: Keywords. |
| Datasheet ID | char | Wahapedia stable datasheet id (per D37). Consistent across a unit's model-group rows. Carried into `units.json` as `unit_id`; where absent (Gen-1 sources, e.g. Chaos Daemons) the converter emits a deterministic `local:<army>:<unit>` fallback. This is the durable saved-list reference key — it survives renames and disambiguates same-named distinct datasheets. Also the key for `unit_loadouts.json` definitions. |

## Tab 2: Unit_Weapons

One row per weapon per Model Group per unit. Covers both base equipment and optional weapons. BS applies to ranged weapons; WS applies to melee weapons — leave the inapplicable field blank.

Weapon ability names are stored as the full parameterized string as printed on the datasheet (e.g. "MELTA 2", "RAPID FIRE 1", "ANTI-VEHICLE 4+"). Generic descriptions are looked up from Tab 7 by the base ability name.

| Field Name | Data Type | Notes |
|---|---|---|
| Army Name | char | **Merge key.** |
| Unit Name | char | **Merge key.** |
| Model Group | char | **Merge key.** |
| Weapon Type | char | Ranged or Melee. |
| Weapon Name | char | As printed on datasheet. Profile-suffixed where multiple profiles exist: e.g. "Plasma pistol – standard", "Plasma pistol – supercharge". |
| Range | char | e.g. 24" or Melee. |
| A | char | Attacks. e.g. 2 or D6. Char to handle dice expressions. |
| BS | char | Ballistic Skill. e.g. 3+. Ranged weapons only; blank for melee. |
| WS | char | Weapon Skill. e.g. 3+. Melee weapons only; blank for ranged. |
| S | int | Strength. |
| AP | int | Armour Penetration. e.g. -1. Use 0 for no AP. |
| D | char | Damage. e.g. 1 or D3 or D6+1. Char to handle dice expressions. |
| Weapon Ability Names | char | Comma-separated parameterized ability names as printed on the datasheet. e.g. "MELTA 2, HEAVY". Full generic descriptions in Tab 7: Weapon_Abilities. |
| Weapon Keyword Names | char | Comma-separated keyword names. Full descriptions in Tab 6: Keywords. |
| Is Base Equipment | char | Yes/No. Yes = part of standard loadout. No = optional substitution weapon. Used by the loadout rollup to seed default weapon counts. |

## Tab 4: Unit_Points

One row per unit per army. Stores model-size options and points costs as published in the GW Munitorum Field Manual. Updated separately from other unit data (~12-13 times per year). Merge key is Army Name + Unit Name only — no Model Group, as points are priced at the unit level.

**Chapter-variant pricing**: When a chapter-specific unit has different points from the generic Adeptus Astartes version, an additional row is added with the chapter name as Army Name. The app looks for a faction-specific row first; if none exists, it falls back to the Adeptus Astartes row.

**Name overrides**: `mfm_points_parser.py` carries a `POINT_NAME_OVERRIDES` dict to handle name mismatches between the MFM and datasheet (e.g. "Myphitic Blight-Haulers" → "Myphitic Blight-hauler") and to suppress Legends units (mapped to None = skip). Update this dict when adding a new faction.

| Field Name | Data Type | Notes |
|---|---|---|
| Army Name | char | **Merge key with Unit_Stats.** |
| Unit Name | char | **Merge key with Unit_Stats.** |
| Size_1 | int | First (smallest) valid model count. Always populated. |
| Size_2 | int | Second valid model count. Blank if the unit has only one size. |
| Size_3 | int | Third valid model count. Blank unless the unit has three size brackets. |
| Points_S1_U1 through Points_S3_U3 | int | 3×3 matrix: rows = size slots, columns = copy tiers (1st / 2nd / 3rd+). |

## Tab 10: unit_loadouts.json (structured loadout definitions)

**This is the primary wargear authority for all units with a definition.** Generated by `loadout_parser.py`, with hand-authored entries preserved and never overwritten. Keyed by `unit_id` (Wahapedia datasheet id).

Each entry has:
- `size_brackets` — ordered list of valid unit sizes, e.g. [5, 10].
- `model_groups` — ordered list of model group objects, each with `name`, `count` (either `{fixed: N}` or `{fills_to_size: true}`), and `default_weapons` (list of base weapon names). Optional per-group fields (see below): `default_wargear`, `default_weapon_counts`.
- `options` — ordered list of option objects.
- `_parser_flags` — present on `loadout_parser.py`-generated entries only; lists sentences that were unmatched or need review. Absent on hand-authored entries.
- `_defaults_source` — set to `"equipped"` on entries whose per-group defaults were written by `equipped_parser.py` from the datasheet "equipped with" wording (D49).

**Per-group default fields (source: `equipped_parser.py`, D49):**
- `default_weapons` — list of base weapon names each model in the group carries. Authoritative source is the datasheet UNIT COMPOSITION "... is equipped with:" prose, which the CSV export drops; the equipped parser recovers per-model attribution the flat pool could not. Variant weapons resolve to all their profiles (e.g. "plasma incinerator" → both "– standard" and "– supercharge").
- `default_wargear` — optional list of non-weapon items from the same prose (Astartes shield, jump pack). Separate from weapons so rollups stay clean (D50). Not yet displayed in the popup.
- `default_weapon_counts` — optional sparse map `{weapon name: N}` for quantities > 1, e.g. a Land Raider's `{"Godhammer lascannon": 2}` (D51). The rollup multiplies by N, defaulting to 1 when absent. Non-breaking.

**Option types:**

| type | What it does | Key fields |
|---|---|---|
| `choice` | Single-model 1-for-1 swap. The user picks one from `choices` or keeps `replaces`. | `scope` (model group name), `replaces` (weapon being dropped), `choices` (list of alternatives) |
| `count` | Size-scaled swap: N models per trigger, max M per trigger. If `replacement_choices` present, the user also picks which weapon each slot uses. | `replaces`, `replacement` or `replacement_choices`, `per_n_models`, `max_per_n` |
| `add` | Model keeps its base weapon AND gains the added weapon. NOT a swap. `blocks_swap: true` reserves that model from the count-swap pool. | `adds_weapon` or `equipment`, `per_n_models`/`max_per_n` or `max_total` |

**Critical distinction (D46):** "equipped with" in GW prose = `add`. "replaced with" = `choice` or `count` swap. Getting this wrong silently produces incorrect weapon counts.

**Shared pool cap:** all `count` options on a body group share one pool (= number of pool-eligible models after `add` reservations). The rollup flags over-allocation but does not block it (D34).

**Parser run command:**
```
python3 loadout_parser.py \
  --options Datasheets_options.csv \
  --units-dir _work/deploy \
  --comp Datasheets_unit_composition.csv \
  --cost Datasheets_models_cost.csv \
  --datasheets Datasheets.csv \
  --factions SM DG \
  --existing _work/unit_loadouts.json \
  --out _work/unit_loadouts.json \
  --report _work/parser_report.md
```

The `--existing` flag preserves hand-authored definitions. Always pass the current `unit_loadouts.json` as `--existing` so hand edits are not lost on re-run.

**Per-model defaults pass (`equipped_parser.py`, D49):** run after `loadout_parser.py`, once per faction, against the same file in and out so both factions accumulate. It keys off `units.json` unit_ids, drops non-roster datasheets, and prunes orphan entries (D52).
```
python3 equipped_parser.py --composition Space_Marines_web.txt --units units.json --loadouts unit_loadouts.json --out unit_loadouts.json --report sm_report.md
python3 equipped_parser.py --composition Death_Guard_web.txt   --units units.json --loadouts unit_loadouts.json --out unit_loadouts.json --report dg_report.md
```
The composition `.txt` files are pasted from Wahapedia datasheet pages (whole page, Legends included — banked as source, non-roster units dropped per D53). The report lists tokens routed to `default_wargear`, weapons with quantity > 1, and any unmatched groups/dropped lines. Reports are regenerated each run and are disposable review artifacts, not repo files.

## Session 13 Addendum (July 2026) — Loadout default fields

**`default_wargear` (per model group, in `unit_loadouts.json`):** now carries real data for the first time — Wardens of Ultramar (`storm shield` on Metaurus, `refractor field` on Gaius). Non-weapon "equipped with" items are routed here by `equipped_parser`, kept separate from `default_weapons`. **No popup UI consumer yet** — the data is correct but not displayed.

**`_defaults_source` semantics:** `"equipped"` means `equipped_parser` assigned authoritative per-group `default_weapons`/`default_wargear`/`default_weapon_counts` from pasted composition prose. `None` (or absent) means the entry still holds `loadout_parser`'s flat placeholder (same weapon set on every group). For a multi-model unit, `None` + identical `default_weapons` across groups is the wrong-per-model-count signature (see Decision Log D57) and inflates weapon counts to ×(unit size) in the rollup.

## Session 15 additions — `unit_loadouts.json` count & choice shapes

**Model-group `count` shapes** (a group's `count` object is exactly one of):
- `{"fixed": N}` — a constant count (leaders, fixed extras).
- `{"fills_to_size": true}` — the body; takes the remainder after all other groups are resolved.
- `{"optional": true, "max": N}` — an optional group (composition row `0-N`). Renders as a toggle (max 1) or a 0-to-N stepper. **Interim (v5.27/5.28):** the pane treats optional groups as **0** until the priced-toggle UI lands.
- `{"per_bracket": {"<size>": N, ...}}` — a count that differs per size bracket (D62, "OR" alternative-profile units). The pane reads the entry for the selected size; non-editable, no stepper.

**Compound choices (D63):** a `choice`-type option's `choices` array may contain a compound entry rendered `"Weapon A + Weapon B"` (two weapons in one pick). Consumers must split a selected choice on `" + "` and tally each weapon. Each weapon in a compound is normalised independently; a part that isn't a weapon (e.g. a shield) is left as-is and flagged `WEAPON_NOT_FOUND ... (compound choice)`.

## Session 15 correction — where sizes/points actually live
Sizes and points are stored under each unit's **`points`** object in `units.json` (`points.sizes` = array of `{size, first_unit, second_unit, third_plus}`; plus the 3×3 points matrix). There is **no** top-level `sizes` array. A unit **without** a `points` object is un-onboarded (no size selector). Points are produced by `mfm_points_parser` from the MFM `.txt` files — **not** by `convert_to_json` / `Unit_Points.csv`, which is stale and must not be used to regenerate.


## Session 16 additions — shared-pool & prerequisite option fields (v5.29)
Two optional fields on `unit_loadouts.json` option objects, added for the Intercessor Sergeant grenade launcher (D64/D65). Both generalise without new code.

- **`pool_id`** (on `add` / `count` options): options sharing the same `pool_id` draw from **one unit-wide cap that may span multiple model groups** (e.g. a Sergeant toggle + a body stepper sharing a per-5 grenade-launcher pool). The cap is defined by whichever pooled option carries `per_n_models`; other pooled options only consume from it, in model-group order (leader first). This is **distinct** from the body-group shared pool of D34, which is per-group. The rollup exposes each shared pool as `pools["pool:<id>"] = {pool, used, leaderUsed}`; a body stepper's live ceiling is `pool − leaderUsed`. Over-allocation is flagged, not blocked.
- **`requires_weapon`** (on an option): the option is legal only while the named weapon is still equipped on its model — i.e. no active `choice` in the same `scope` has replaced it. If the prerequisite breaks, the control **disables and auto-clears** (freeing any pool slot). Hard interlock per D64. Enforced in both the renderer and the rollup engine.
  **D103:** the value may name **several** weapons joined with ` + ` (e.g. `Heavy bolt pistol + Astartes chainsword`); the bearer must hold **every** one of them. This is the same ` + ` convention used by `replaces`, `replacement` and `choices` — no separate field. *Engine note: v5.56 reads the value as a single name, so a compound gate is currently over-strict (the option never enables). B32.*

## Session 17 additions — any-number cap fields & pool semantics (v5.32)
- **`max_total_all`** (bool, on `count` options): marks an "any number / all models / up to N" swap. The per-option ceiling is resolved **at render time** to the scoped model group's model count — `loMaxCount(opt, size, groupN)` returns `groupN` (all eligible models), or `min(up_to, groupN)` when `up_to` is set. Replaces the old behaviour where these options had no usable cap (showed "max 0"). The former `REVIEW_MAX_TOTAL` parser flag is retired.
- **`up_to`** (int, optional, on `max_total_all` options): the N from an "Up to N models can…" line. Enforces the ceiling (e.g. Devastator = 4, Deathwatch Terminator = 3). Absent for "Any number"/"All" (unbounded up to group count).
- **Per-source-weapon pool (rollup):** a model may swap **each weapon it carries once**, so the binding limit is per source weapon, not the sum of all body swaps. The rollup charges each named source weapon (compound sources like `Power fist + Storm bolter` charge each part) and flags over-allocation when any single source weapon is swapped more than the group has models. `pools[<group>] = {pool, used}` where `used` is the most-consumed source weapon. Independent swaps (different sources on the same models) no longer false-warn; swaps sharing a source stay coupled.
- **Compound weapons (`" + "`)** now appear on both sides of `count` options: `replaces` may be `"A + B"` (a swap that removes both) and `replacement`/`replacement_choices` entries may be `"A + B"` (adds both). The rollup splits on `" + "` for source removal and replacement addition.
- **Source-derived `group` label:** any-number swaps use `"<Source Weapon> Options"` as their sub-heading (e.g. `"Power Fist Options"`) instead of the generic `"Special Weapon"`, so distinct swaps render as separate groups. Per-N special-weapon options keep `"Special Weapon"`. `group` is presentation only — no rollup/keying logic depends on it.

- `unit_ability_details` — map of ability name → this unit's own description. The render

## Session 19 additions — bundle grants, `loadout_relation`, characteristic overrides (v5.42)

**`bundled_swaps.json` grant fields** (authored source; passed through to `units.json`
by `convert_to_json`, attached to a unit as `bundled_swaps[].endpoints[].grants[]`):
- `option_name` / `ability_name` — the wargear granted and the ability it confers.
- `carrier_notes` — the ability's rules text (e.g. "The bearer has a Wounds
  characteristic of 6."). This is now the primary source the render reads a
  characteristic effect from (see below).
- `stat_override` — a map of stat key → value (`{"W":"6"}`, `{"INV":"4"}`,
  `{"SV":"3"}`, `{"FNP":"6"}`). Explicit escape hatch: it wins over the reader when
  present, for text the reader can't parse. Values are bare numbers as strings; the
  stat table appends the trailing `+` for SV/INV/FNP.

**`loadout_relation`** (on each `bundled_swaps` group; default `"owns"`):
- `"owns"` — the bundle owns loadout options it covers; those options are suppressed in
  the loadout UI and the bundle picker drives them (e.g. Captain).
- `"alternative"` — the bundle is an atomic all-or-nothing alternative loadout that
  cross-excludes the per-slot loadout swaps; picking it hides those swaps (e.g.
  Lieutenant 3-for-3).

**Characteristic-from-text reader (v5.42, B12 remainder).** The render derives stat
overrides from ability text via a general reader, replacing per-effect hardcoding. It
recognizes the fixed 40K phrasings: `N+ invulnerable save` → INV, `Wounds
characteristic of N` → W, `Save characteristic of N+` → SV, `Feel No Pain N+` → FNP.
Sources read: (1) active bundle-grant `carrier_notes` (authored `stat_override` wins);
(2) active weapon/wargear abilities via the weapon-ability glossary — this de-hardcodes
the former Shining Aegis special-case (Keeper of Secrets → Sv 3+). The weapon/wargear
path deliberately skips any ability already surfaced through `wargear_ability_names`
(the "broad" conferred-wargear surface), which is handled by its own pass (B12-broad /
E17) so bearer-only and conditional effects are not mis-applied to the group statline.

## B14 update (v1.5) — Other Options now carry optional wargear items

`other_options` (previously empty for SM) now holds optional per-model wargear-item
adds parsed from "1 X can be equipped with 1 Y":

- `option_name` — the proper wargear-ability name (footnote asterisk stripped).
- `max_per_unit` — the "1 Y" count (currently always 1).
- `carrier_notes` — the item's ability text **only when it is an unconditional
  set-value characteristic** (e.g. Helix Gauntlet's "Feel No Pain 6+"); blank when the
  set-value sits inside a conditional/summon clause (Watcher in the Dark) so the reader
  can't misfire. Non-stat items keep their descriptive text here for display; the
  reader ignores it.
- `ability_name` — same proper name; the glossary definition backs the tooltip.
- `exclusion_group` + `is_mutually_exclusive="Yes"` — set for item-only "one of the
  following" lists; the app treats a shared `exclusion_group` as a radio. Independent
  single adds carry no group.

A routed item's ability is **removed** from the unit's always-on `wargear_ability_names`
(it now confers only when the option is checked). Base-equipment wargear abilities are
untouched (e.g. Spectrus keeps Instigator Bolt Carbine).

## `unit_loadouts.json` — weapon naming rule (D95, restated D96)

Every weapon name held in `unit_loadouts.json` is a **base / family name**. A profile suffix
("Plasma pistol – standard", "Bilesword – strike") never appears — not in `default_weapons`, not in
`default_weapon_counts` keys, and not in an option's `replaces`, `replacement`, `choices`,
`replacement_choices`, `adds_weapon` or `requires_weapon`. The app treats the family, not the
profile, as the unit of selection, replacement and display, and looks every weapon up by base name;
a profile suffix left in place makes a swap consume only one profile of the weapon it is meant to
remove. `loadout_parser.py` emits base names, and a `normalise_profiles` pass in `equipped_parser.py`
folds the whole file on every run, so the invariant cannot drift.

`' + '` is **not** a profile suffix: it joins a compound. A compound may appear on either side of a
swap — `replaces: "Boltgun + Power weapon"` (the model gives up both) and
`replacement: "Frag cannon + Close combat weapon"` (the model gains both). The rollup splits on
`' + '` and charges/grants each named weapon. As of D96 an option's display heading (`group`) is also
derived from the base name, so it carries no profile suffix either.

**Per-bracket swaps.** `per_n_models: N` with `max_per_n: M` means "for every N models in the unit, up
to M models may take this swap". `M` is read from the datasheet sentence, not assumed to be 1.

## Session 42 additions (v1.7) — `equipment_choices`, compound sources, absent sources (D97)

**`equipment_choices` (new, `choice` options only).** An add-choice option (`type: "choice"` with
`replaces: null` — "this model can be equipped with one of the following") may offer picks that aren't
weapons: a shield dome, an orbital comms array, an auspex array. Those names are listed in
`equipment_choices`, a subset of `choices`. The engine files a pick named there under **equipment**;
anything else goes to the **weapon** table. The parser fills the field by matching a pick that failed
the weapon lookup against the wargear allowlist in `weapon_abilities.json`, tolerating a trailing
qualifier the datasheet omits ("Orbital Comms Array (Aura)" → "Orbital Comms Array"). Absent when no
pick is an item. Carriers today: both Impulsors, the Corvus Blackstar.

**Compound source (`replaces: "A + B"`) — now legal everywhere.** Since D97 the engine consumes a
compound source on a **single-model group** as well as a multi-model one: every weapon named is removed
and the replacement is emitted once. As of D98 the parser writes the full source on `choice` and
`single` options too, splitting the datasheet phrase on commas and " and " ("bolt pistol, master-crafted
bolter and close combat weapon" → `"Bolt pistol + Master-crafted bolter + Close combat weapon"`).
`COMPOUND_SOURCE_UNSUPPORTED` and `COMPOUND_SOURCE_ON_SINGLE_GROUP` are retired. The option's `group`
heading is taken from the **first** source weapon, so a compound never reaches a heading.

**`OR_SOURCE_UNSUPPORTED` (new flag).** A source phrase that offers an alternative inside a compound
("its Fenrisian greataxe **or** great wolf claw **and** storm bolter can be replaced…") names a weapon
the model may not have. The engine keys sources by name and has no "either of these" source, so the
parser flags the sentence and falls back to the first source weapon. One carrier: the Wulfen
Dreadnought.

**Quantity qualifier on an add ("1 additional combi-bolter").** "additional" / "another" / "second" /
"extra" is not part of a weapon's name. The parser strips it and resolves the weapon, so the add stacks
a second copy of a weapon the model already carries rather than inventing one under a literal name
(D98). Carriers: Chaos Rhino, Helbrute.

**Absent source = inert option.** An option whose `replaces` names no weapon that its scope group
actually carries is skipped by the rollup and hidden by the UI. This is the safe reading of two known
data faults: a variant-group swap parked on the base group (B21), and a swap whose source is a wargear
*item* rather than a weapon (death totem, storm shield). Fixing either fault re-activates the option
with no engine change.

**Mutually exclusive choices.** Two `choice` options in the same `scope` whose source weapons overlap
are alternatives for one slot. The app renders them as a single radio group; nothing in the data marks
the relationship — it is derived from `replaces`.

## Session 44 additions (v1.9) — `equipment_parts` (D99)

**`equipment_parts` (new, any swap option: `choice`, `count`, `single`).** A list of names which, wherever
they appear on the **replacement side** of that option, are filed under **equipment** rather than the
weapon table. It applies to a name standing alone and to a named part of a `' + '` compound, and it covers
every replacement shape: `choices`, `replacement_choices`, and a single `replacement`.

Example (authored shape, lands with B30): the Captain with Jump Pack's option replaces
`"Heavy bolt pistol + Astartes chainsword"` with the single choice
`"Thunder hammer + Relic shield"` and carries `equipment_parts: ["Relic shield"]`. The rollup emits a
thunder hammer to the weapon table and a relic shield to "Other wargear".

**Relationship to `equipment_choices` (D97).** `equipment_choices` names *whole picks* on an add-choice
(no source replaced); `equipment_parts` names *parts of a replacement* on a swap. The engine reads both
into one per-option set and routes any matching name to equipment, so the two never conflict and a name
may sit in either. `equipment_choices` is retained as-is; existing carriers (both Impulsors, the Corvus
Blackstar) are unchanged.

**Validity.** A name listed in either field is a **real** pick even though no weapon profile exists for it.
The UI's broken-choice test treats it as real, so an option whose only pick is an item is shown, not
dropped.

**Absent** when an option's replacements are all weapons — which is every shipped option as of v5.55.

## Session 45 additions (v2.0) — how the parser emits `equipment_parts` (D100)

The field itself is unchanged from v1.9. What is new is that the parser now writes it, and there is one
convention worth pinning down.

**Where item names come from.** An item name is written in the **display form held in the wargear
allowlist** (`weapon_abilities.json`), not in the datasheet's own casing. So the datasheet's "1 relic
shield" is emitted as `Relic Shield`, and the same string appears in the option's `choices` /
`replacement_choices` / `replacement` and in its `equipment_parts` list. The two must always match, since
the engine keys the routing decision on the name.

**Which side of an option is routed.** Only the **replacement** side. A name on the **source** side
(`replaces`) that is a wargear item rather than a weapon is a separate, still-open problem (B28) and is
still flagged `WEAPON_NOT_FOUND`. The parser deliberately does not consult the allowlist for `replaces`.

**Which shapes are routed.** All replacement shapes: a `choice` list, a `count` option's `replacement` or
`replacement_choices`, and a `single` swap's replacement — including a named part of a `' + '` compound.

**A single replacement can now be a compound.** A `single`-type sentence whose replacement names several
things ("1 thunder hammer and 1 relic shield") is split on `" and "` / commas the way the choice-list side
already was, and written as one compound pick (`Thunder hammer + Relic Shield`). The whole phrase is left
alone when it is itself a known weapon ("Slaughter and Carnage").
